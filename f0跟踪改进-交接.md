# f0 心率跟踪改进 —— 交接文档

> 本文记录 2026-06-13 对 `CombExtractor` 心率基频 f0 估计的改进:**HPS 谐波评分 + 时间连续性惩罚 + 重捕获**。
> 配套主文档:[`交接文档.md`](交接文档.md)、[`梳状提取-改进方向.md`](梳状提取-改进方向.md)。

---

## 1. 为什么要改(根因)

整个上位机的信号质量瓶颈追溯到同一个源头:**f0 不稳定**。

```
f0 跳变(如 0.75↔1.375Hz 之间反复)
  → 梳状滤波器锁在错误频率
  → 提取通道输出的是错误频率的噪声
  → SyncAverager 收到噪声而非心拍
  → 模板无法收敛 / 频繁重置
  → 平均脉搏波质量差
```

老算法在 [0.9,2.2]Hz 内直接 `argmax(Welch谱)` 取最强峰,**每次独立估计、无历史记忆**,所以会从 82bpm 一跳跳到 45bpm。

---

## 2. 改了什么(三层)

代码全部在 `server.py` 的 `CombExtractor._estimate()` 里,逐层叠加:

### 第一层:HPS 谐波加权评分(替代 argmax)
对每个候选频率 f 算谐波得分:
```
Q(f) = SNR(f)×1.0 + SNR(2f)×0.5 + SNR(3f)×0.25
```
三个细节防止"半频/倍频锁定":
- **窗口收窄** ±0.10Hz(`HW=0.10`):防止相邻强峰被误算成谐波。
- **局部噪声底**:每个谐波位置单独估噪声(fc±0.5Hz 去掉峰窗的中位数),下限 = 全局中位数×0.5,防止高频稀疏区 SNR 虚高。
- **邻近度权重**:峰离 k·f 越远贡献越小(`proximity = 1 - |peak_freq - fc|/HW`)。

### 第二层:时间连续性惩罚
归一化 Q 后,减去偏离上一次 f0 的代价:
```
Q_eff(f) = Q(f)/Qmax - w_cont × |f - f0_prev|
```
`w_cont` 按当前置信度分档(`_CONT_W`):

| 状态 | w_cont | 行为 |
|------|--------|------|
| 冷启动 (f0=None) | 0 | 自由捕获,第一帧纯看 Q |
| conf=0 (极低) | 0.3 | 弱惩罚,基本放开重捕获 |
| conf=1 (低) | 0.8 | 中等,抑制小跳变 |
| conf=2 (可用) | 1.5 | 强锁定,压住半频/倍频跳变 |

**w 推导**(以 0.75 vs 1.375Hz 为例):归一化后 Q(0.75)=1.0、Q(1.375)=0.11,Δf=0.625。conf=2、w=1.5、上帧 f0=1.375 时:
```
s(0.75)  = 1.0  - 1.5×0.625 = 0.06
s(1.375) = 0.11 - 0          = 0.11  → 选 1.375 ✓
```

### 第三层:重捕获逃生阀(防锁死)
连续性惩罚的风险是**锁死在错误频率**(Codex 评审重点警告:稳定地错时 std 也低、conf 也高,会自我强化)。所以加逃生:
```
原始最优 raw_f(不带惩罚)持续偏离当前 f0 超过 _RECAP_DF(0.20Hz)
  → _challenge 计数累加
  → 累计 _RECAP_N(6 帧 ≈ 3s) → 强制 best_f = raw_f, 清空 f0 历史重评置信度
```
这给了系统**自愈能力**:即使第一帧选错或长期锁错,只要真实心率持续在别处,3s 后会强制跳轨纠正。

## 3. 关键参数(类常量,在 `CombExtractor` 顶部)

| 参数 | 值 | 含义 |
|------|----|----|
| `hr_lo, hr_hi` | 0.7, 2.5 | f0 搜索带(Hz) |
| `HW`(局部变量) | 0.10 | 谐波窗半宽(Hz) |
| `_STD_OK` | 0.20 | f0_std < 此值 → conf=2 |
| `_STD_LOW` | 0.35 | f0_std < 此值 → conf=1 |
| `_CONT_W` | {0:0.3, 1:0.8, 2:1.5} | 连续性惩罚权重(按 conf) |
| `_RECAP_DF` | 0.20 | 重捕获:raw 偏离阈值(Hz) |
| `_RECAP_N` | 6 | 重捕获:累计帧数(每帧 ~0.5s) |
| `_F0_HIST_LEN` | 8 | f0 历史队列长度(算 std 用) |

新增状态变量:`self._challenge`(重捕获计数)。

## 4. 实测结果(三段真实录制)

模拟实时逻辑(16s 窗、每 0.5s 估一次)在 `recordings/` 上的 f0_std:

| 录制 | 改前 std | 改后 std | 结论 |
|------|---------|---------|------|
| rec_20260612_221451 | 0.264 Hz | **0.129 Hz** | 砍半,锁稳在 1.37Hz |
| rec_20260612_225242 | — | **0.166 Hz** | 稳定 |
| rec_20260612_224518 | 0.396 Hz | **0.419 Hz** | 几乎无变化 |

**关键认知**:rec_224518 改后仍不稳,是因为**该段原始信号本身没有稳定主峰**(raw_f 自己每十几秒就换频率,在 0.75/1.375/2.5Hz 之间游走)。连续性惩罚和重捕获都正常工作,但它们只能平滑"算法噪声",**不能创造数据里不存在的稳定性**。这是正确行为——不应强行让噪声"看起来稳定"。

## 5. 怎么用 Q 和 f0_std 判断信号好坏

这两个数字是判断"现在测的信号能不能用"的客观依据,比红/黄/绿三档更细。

**Q 分(谐波得分)—— 看单帧信号强不强:**
| Q | 含义 |
|---|------|
| > 30 | 信号很好,谱峰清晰(好段有 45~112) |
| 10~30 | 可用 |
| < 10 | 信号弱,接近噪声 |
| < 5 | 基本在噪声里猜 |

**f0_std —— 看跟踪稳不稳:**
| f0_std | 含义 |
|--------|------|
| < 0.10Hz | 锁得很稳 |
| 0.10~0.20Hz | 可用(conf=2 绿) |
| 0.20~0.35Hz | 不稳(conf=1 黄) |
| > 0.35Hz | 锁不住(conf=0 红) |

**两个一起看(关键):**
```
Q高 + std低  → 信号真好,放心用 ✓
Q高 + std高  → 有强信号但多频打架(如 0.75/1.375)
Q低 + std低  → ⚠️危险! 锁得稳但锁的是噪声(假稳定, 颜色会显绿但其实没信号)
Q低 + std高  → 信号差,如实反映 ✓
```
> 最需警惕 **"Q低+std低"**:置信度颜色显绿(只看 std),但 Q 揭示真相。rec_224518 后半段就是这种假稳。

**测量时用法**:调整探头位置时盯着 Q,Q 跳上去 = 这个位置接触好了。

## 6. 局限与下一步

- **本方案是单步贪心 + 重捕获**,不是 Codex 推荐的 Viterbi 全局最优。区别:Viterbi 能同时保留多条假设轨迹、回溯选整条最优;贪心跳错只能靠 ~3s 重捕获纠正。简单可靠、状态少,对本应用够用。若发现重捕获太慢/误跳,再升级 Viterbi。
- **对本身无周期性的弱信号无能为力**——根治要靠硬件端改善接触/光路/前端 SNR(见 `梳状提取-改进方向.md` 方向6)。
- **未做**:Q 分 / f0_std 数值还没显示到网页侧栏(目前只有红/黄/绿三档颜色 + 心率)。下一步可在"信号状态"区加一行 `Q: 12.3 / std: 0.18Hz`,让实测时直接看到信号质量。
- 配套的**梳状带宽自适应**(f0 稳时收窄齿宽到 ±0.12Hz)也未做,依赖 f0 先稳——本次改动是其前置条件。
